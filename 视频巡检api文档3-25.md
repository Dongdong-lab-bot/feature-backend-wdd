# 📘 智慧食安 - 视频巡检模块 API 契约 (V1.0.0)

**适用端**: 监管端 Web/App、执行端 Web/App
**接口风格**: RESTful + 统一响应结构
**依赖说明**: 本模块依赖【台账模块】的基础数据（区域、食堂、人员等）以及【视频监控模块】的设备数据。

---

## 0️⃣ 全局规范 (Global Specs)

### 0.1 统一响应结构
```json
{
  "code": 20000,
  "message": "success",
  "data": {},
  "request_id": "trace-abc123"
}
```

### 0.2 核心业务状态码
| code | 说明 | 处理建议 |
| :--- | :--- | :--- |
| **20000** | 成功 | - |
| **40009** | 状态非法 | 提示“当前状态不允许执行此操作”，刷新列表 |
| **40901** | 并发冲突 | 提示“整改版本已更新”，强制刷新页面 |
| **40401** | 路由匹配失败 | 提示“未知的巡检类型，无法匹配审核流” |

### 0.3 时间与幂等
* **时间**: 统一使用 **UTC ISO8601** (e.g., `2026-03-05T12:00:00Z`)。
* **幂等**: 提交类接口强制要求 Header 携带 `Idempotency-Key: uuid`，防止重复提交。

### 0.4 外部模块依赖
*   **组织与食堂层级树**: 本模块强依赖【用户中心/台账模块】提供的组织与食堂层级树接口。
    *   **响应**: 返回标准的树状结构，每个节点包含 `id`, `name`, `children`。

## 核心枚举与流程图 (Enums & Flow)

### 核心字典字典配置
* **`inspection_type` (巡检类型)**: 
  * `DAILY` (日管控)
  * `WEEKLY` (周排查)
  * `JOINT` (联合巡检)
  * `VIDEO` (视频巡检)
* **`status` (业务状态)**: 
  * `PENDING` (待上报 / 待签字)
  * `SUBMITTED` (已上报，待审核)
  * `REJECTED` (审核驳回，待整改)
  * `RECTIFIED` (已整改，待复审)
  * `COMPLETED` (已完成)

### 状态流转图 (State Diagram)
> 💡 注：不同 `inspection_type` 会动态路由到不同的审核流节点，但基础状态机保持一致。

```mermaid
stateDiagram-v2
    [*] --> PENDING: 模板下发/定时生成

    state 执行端操作 {
        PENDING --> SUBMITTED: 填报并提交
        REJECTED --> RECTIFIED: 提交整改(携带当前版本号)
    }

    state 监管端审核 {
        SUBMITTED --> COMPLETED: 审核通过
        SUBMITTED --> REJECTED: 审核驳回
        RECTIFIED --> COMPLETED: 复审通过
        RECTIFIED --> REJECTED: 复审驳回
    }

    state 联合巡检专属 {
        PENDING --> COMPLETED: 参与多方全部签字完成(系统自动闭环)
    }
```

### 图片来源 (EvidenceSource)

用于在提交时标记照片来源，确保证据链权威性。

*   `USER_UPLOAD`: 用户相册上传
*   `CAMERA_CAPTURE`: 用户实时拍摄

### 1.1 播放参数接口（实时/回放监控画面）
- Path（建议）：POST /api/v1/video/hikvision/play-params

请求体：
```json
{
  "cameraId": 123,
  "action": "preview",
  "begin": "2026-03-21T08:00:00+08:00",
  "end": "2026-03-21T09:00:00+08:00"
}
```
字段说明：
- `cameraId`：业务摄像头/点位 ID
- `action`：`preview` 或 `playback`
- `begin` / `end`：回放必填；预览可不传（建议用 ISO8601）

响应体（Web/UIKIT，`X-App-Client: web`）：
```json
{
  "code": 20000,
  "msg": "success",
  "data": {
    "uikitAccessToken": "...",
    "deviceSerial": "D20591677",
    "channelNo": "1",
    "begin": "20260321080000",
    "end": "20260321090000"
  }
}
```

响应体（App/SDK，`X-App-Client: app`）：
```json
{
  "code": 20000,
  "msg": "success",
  "data": {
    "oauthToken": "...",
    "deviceSerial": "D20591677",
    "channelNo": "1",
    "validCode": "ABCDEFG"
  }
}
```

失败示例（设备加密未关且无验证码可用）：
```json
{
  "code": 40000,
  "msg": "device encrypted: please disable encryption or provide validate code",
  "data": null
}
```

### 1.2 抓拍监控画面
*   **`POST /video-inspections/cameras/{camera_id}/capture`**
*   **场景**: 监管人员发现视频中有违规行为时，前端通过 Canvas/UIKit 截取当前画面，上传 Base64 数据保存为图片，用于后续作为巡检记录的图片证据。
*   **Request**:
```json
{
  "image_base64": "data:image/jpeg;base64,/9j/4AAQSkZJRg...", // 前端截取的画面 Base64 数据
  "timestamp": "2026-03-17T10:00:00Z"
}
```
*   **Response**:
```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "photo_url": "https://cdn.safefood.com/captures/20260317_cam001_abc.jpg" // 返回存到OSS/服务器的图片URL，供填表时使用
  }
}
```

## 2️⃣ 模板管理 (`/video-inspections/templates`)

> **业务核心**: 监管端创建视频巡检表格，与周排查类似，但检查小项支持“关联摄像头”功能。

*   **`GET /video-inspections/templates`**: 获取视频巡检模板列表。
*   **`POST /video-inspections/templates`**: 创建新的视频巡检模板。
*   **`GET /video-inspections/templates/{id}`**: 获取单个模板详情。
*   **`PUT /video-inspections/templates/{id}`**: 更新指定模板。
*   **`PATCH /video-inspections/templates/{id}/status`**: 快速启用/禁用模板。
*   **`DELETE /video-inspections/templates/{id}`**: 删除指定模板。

**[POST/PUT] Request Body 示例 (包含关联摄像头)**:
```json
{
  "template_name": "食堂视频巡检检查表",
  "executor_role": "FOOD_SAFETY_DIRECTOR",
  "approver_role": "SUPERVISOR",
  "target_node_ids": ["region_1", "canteen_102"],
  "start_time": "06:00",
  "end_time": "20:00",
  "form_type": "SCORE_BASED", // 表格类型: 选分表
  "major_items": [
    {
      "title": "食材问题排查",
      "minor_items": [
        {
          "content": "食堂无三无、腐烂、过期食材",
          "issue_type": "RED_LINE",     // 问题类型：红线/黄线等
          "total_score": 6,             // 总分
          "scoring_options": [6, 3, 0], // 打分项设置
          "associated_camera_ids": ["camera_001", "camera_002"] // 【差异点】关联的摄像头设备ID列表
        }
      ]
    }
  ]
}
```

### 3.3 监管端提交检查结果 (填表&提问题)
*   **`POST /video-inspections/tasks/{id}/submit`**
*   **场景**: 监管端人员通过视频监控抓拍并记录违规问题。
**Request Body**:
```json
{
  "inspector_id": "user-regulator-wangwu",
  "actual_start_time": "2026-03-17T09:30:00Z",
  "results": [
    {
      "item_id": "uuid-item-video-1",
      "score_given": 0, 
      "description": "通过监控发现桌面有大量未清理餐具",
      "photos": ["https://.../capture_photo.jpg"] // 此处的照片通常来源于抓拍接口的返回
    }
  ]
}
```

### 3.4 食堂端提交整改
*   **`POST /video-inspections/tasks/{id}/rectify`**
*   **场景**: 食堂端针对视频巡检提出的问题进行线下整改并上传反馈照片。
**Request Body**:
```json
{
  "rectifier_id": "user-canteen-zhaoliu",
  "feedback_per_item": [
    {
      "result_id": "uuid-result-video-1",
      "description": "已安排保洁人员对前厅桌面进行全面清理消毒",
      "photos": ["https://.../rectify_clean.jpg"]
    }
  ]
}
```

### 3.5 监管端审核整改
*   **`POST /video-inspections/tasks/{id}/audit`**
*   **场景**: 监管端对食堂提交的整改内容进行审核，通过则流程闭环，驳回则打回继续整改。
**Request Body**:
```json
{
  "auditor_id": "user-regulator-wangwu",
  "action": "PASS", // PASS (通过) | REJECT (驳回)
  "opinion": "桌面已清理干净，整改合格"
}
```
