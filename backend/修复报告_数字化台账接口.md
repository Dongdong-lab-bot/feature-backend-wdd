#### 问题描述

- **影响接口**: `GET /ledger/instances`
- **根本原因**: API 层代码访问了 `record.created_at`，但模型中定义的是 `create_date` 和 `create_time`

#### 修复方案
修改 `backend/app/modules/ledger/api.py` 第 57 行：

```python

# 修复后
"created_at": record.create_time or record.create_date,
```

### 问题 3：台账验证逻辑分析

#### 问题描述
- **现象**: 只有一条数据能返回 `is_valid: true`，其余均返回 `false`
- **影响接口**: `GET /ledger/instances/{id}/verify`

#### 分析结论

验证逻辑本身**没有问题**，验证失败的原因如下：

1. **验证原理**：使用 SHA256 哈希算法验证台账内容完整性
   - 提交时计算：`security_hash = SHA256(content + schema_snapshot + salt)`
   - 验证时计算：`current_hash = SHA256(content + schema_snapshot + salt)`
   - 如果内容被篡改，哈希值不匹配，验证失败

2. **验证失败的可能原因**：
   - 台账数据在提交后被修改（内容不一致）
   - 使用 draft 接口暂存的数据未经过签名验证
   - 测试数据本身就不是有效的已签名台账

#### 验证逻辑代码

**service.py - verify_ledger 方法**：
```python
@staticmethod
def verify_ledger(db: Session, ledger_id: int) -> bool:
    ledger = LedgerService.get_ledger(db, ledger_id)
    current_hash = calculate_security_hash(ledger.content, ledger.schema_snapshot, SALT)
    return current_hash == ledger.security_hash
```

**security.py - calculate_security_hash 函数**：
```python
def calculate_security_hash(content: Dict, snapshot: Dict, salt: str) -> str:
    raw = json.dumps(content, sort_keys=True, separators=(",", ":"))
    raw += json.dumps(snapshot, sort_keys=True, separators=(",", ":"))
    raw += salt
    return hashlib.sha256(raw.encode("utf-8")).hexdigest()
```

#### 结论
验证逻辑工作正常，只有已签名的台账才能通过验证。未签名的台账（如草稿状态）验证失败是正确的行为。
