## 1. 🌍 全局规范（Global Standards）

### 1.1 请求头（Request Headers）

| Header | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 是 | `Bearer {accessToken}`。登录、刷新接口除外。 |
| `Content-Type` | 是 | `application/json`（文件上传除外）。 |
| `X-App-Client` | ⚠️ 条件必填 | **仅刷新接口要求**，值需与登录时的 `app_client` 完全一致，例如 `reg_web`。 |
| `X-Global-View` | 否 | 监管大屏专用。后端会再次校验角色，普通用户即便传递也会被覆盖为 `false`。 |

### 1.2 统一响应结构（Response Structure）

```json
{
	"code": 200,
	"msg": "success",
	"data": { "...": "..." }
}
```

- `code`：业务状态码（200 成功 / 400 参数错误 / 401 Token 无效 / 403 无权限 / 500 服务异常）。
- `data`：实际业务负载，列表和分页均包裹在此。

## 2. 🔐 认证鉴权（Auth Module）

> 当前仅开放 **登录 / 刷新 / 权限查询 / 外部 IdP**，`/auth/logout` 与 `/auth/me` 尚未实现，已在需求池排期。

### 2.1 用户登录

- **Method & Path**：`POST /auth/login`
- **请求体**：

```json
{
	"username": "admin",
	"password": "encrypted_base64",
	"app_client": "reg_web"
}
```

- **app_client 取值**：`reg_web` / `reg_app` / `exec_web` / `exec_app`。后端会校验角色与端侧是否匹配，否则返回 `403`。
- **响应体**：

```json
{
	"code": 200,
	"data": {
		"accessToken": "eyJhbGciOi...",
		"refreshToken": "rft_abc...",
		"userInfo": {
			"id": 1001,
			"roleType": "REGULATOR",
			"tenantId": 1
		}
	}
}
```

### 2.2 刷新 Token（防掉线）

- **Method & Path**：`POST /auth/refresh`
- **请求头**：必须携带 `X-App-Client: {与登录一致}`。
- **请求体**：`{ "refreshToken": "..." }`
- **主要校验**：刷新 Token 与 Header 的 `app_client`、`tenantId`、`tokenVersion`、`jti` 必须一致，且用户状态为 `ACTIVE`。
- **响应体**：与登录一致，仅返回新的 `accessToken` / `refreshToken`。

### 2.3 获取当前权限列表

- **Method & Path**：`GET /auth/me/permissions`
- **授权**：需要登录态；服务端会根据用户 ID 查询 RBAC 权限集合。
- **响应示例**：`{"code":200,"data":["ledger:view","inspect:dispatch"]}`。

### 2.4 外部 IdP / OIDC 登录

- **Mock IdP**：`POST /auth/idp/mock-login`（仅非生产环境开启）。
- **OIDC 登录**：`POST /auth/idp/oidc-login`。
- **核心要点**：
	- `app_client` 仍必填，沿用内置四种端标识。
	- 需要提供 `issuer/sub`（mock）或 `id_token/nonce`（OIDC）。
	- 首次登录会自动创建或绑定账号，返回结构同 2.1。

## 3. 🏢 组织架构（Org Module）

### 3.0 概念对齐

- **租户（Tenant）**：跨组织的数据隔离边界，由平台侧创建。
- **组织（Org）**：租户内部的业务树节点（片区、学校、食堂）。

### 3.1 获取组织树

- **Method & Path**：`GET /org/tree`
- **实现现状**：
	- `user/api.py`：根据 `root_id + tenant_id` 查询真实组织树（多租户隔离）。
	- `user/org_api.py`：仍保留 Demo 数据版本，供前端自测使用。
- **下一步**：合并两个实现，统一返回结构，并支持 `type / parentId / lazy` 等筛选。

### 3.2 创建租户（平台 API，Demo）

- **Method & Path**：`POST /tenant`
- **说明**：`org_api.py` 提供的 Demo，未做权限校验与真实落库，需谨慎使用。

### 3.3 创建组织/食堂（租户 API，Demo）

- **Method & Path**：`POST /org`
- **说明**：当前实现为 Mock 数据，后续会接入真实模型与租户上下文。

## 4. 👥 用户管理（User Module）

### 4.1 用户状态变更

- **Method & Path**：`PUT /user/{user_id}/status`
- **权限**：调用者必须拥有 `user:status:update`。
- **请求体**：`{ "status": "ACTIVE" | "DISABLED" }`
- **行为**：禁用用户会自增 `tokenVersion`，导致所有旧 Token 立即失效。

## 7. 💡 对接提示（Critical Tips）

1. **多端隔离**：`app_client` 是所有认证接口的第一道闸门，务必与端侧匹配，否则得到 `403 forbidden`。
2. **刷新 Token**：`X-App-Client` 一定要与登录时的 `app_client` 一致；如前端缓存了多个账号，需要按端隔离存储。
3. **组织树重复路由**：短期内可继续访问 Demo 版 `/org/tree` 做静态演示，但真实联调请切换至 `user/api.py` 版本，并传入 `root_id`。
4. **图片能力**：当前实现仅供 Demo，尚未对 tenant/user 权限做校验，线上前请勿开放。
5. **接口缺口**：文档仍保留 `logout`、`/users` 等章节，但请在使用前务必确认实现状态，以免调用 404。
