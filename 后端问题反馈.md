# 后端问题反馈

本文档记录了在前端联调“日管控”模块时发现的后端接口及数据相关问题，请后端开发人员跟进修复或调整。

## 1. **数据合并 Bug（日管控 & 周排查通用）**：

- 接口：`GET /daily-controls/tasks/{id}` 以及 `GET /weekly-inspections/tasks/{id}`
- 现象：详情接口返回的 `form_snapshot` 数据中丢失了 `is_qualified`（日管控）或 `score_given`（周排查）等执行端填报结果。
- 原因：在 `api.py` 的数据合并逻辑中，从 `result_list` 提取字典时，`result_map` 的 key 是数据库原生的 `int` 类型（`item_id`），但 `form_snapshot` 中的 `minor_data.get("item_id")` 有时被处理成了字符串，或者由于某些其他原因类型不匹配，导致 `result_map.get()` 取不到值，直接造成业务关键字段丢失。
- 前端已做防御性容错（如周排查默认补齐满分，日管控默认补齐合格），但后端仍需排查合并逻辑的类型转换。

**建议后端修改：**
在生成字典和获取键值时，统一强制转换为字符串类型进行匹配即可：

```python
result_map = {str(result.item_id): result for result in result_list}
# ...
result = result_map.get(str(item_data.get("item_id")))
```

## 2. 角色权限拦截抛出 `403 Forbidden`

**现象：**
前端调用 `POST /daily-controls/tasks/{id}/submit`（提交）和 `POST /daily-controls/tasks/{id}/audit`（审核）时，即使使用了系统超级管理员账号（`superadmin`），依然会遇到 `403 Forbidden` 报错。

**原因分析：**

1. **基础权限拦截**：
   这是正常的权限拦截逻辑：

   ```python
   await _ensure_permission(db, current_user, DAILY_SUBMIT, "无提交权限")
   await _ensure_permission(db, current_user, DAILY_APPROVE, "无审核权限")
   ```

   只要登录账号在数据库的 `role_permissions` 表中没有被配置 `daily:submit` 或 `daily:approve` 权限，就会被正确拦截。
   但经测试发现，即使在数据库中通过脚本强制为 `superadmin` 用户绑定了拥有所有权限的角色，后端的 `_ensure_permission` 仍可能返回 `False`，说明后端权限鉴权逻辑或权限缓存数据存在问题。
2. **`inspector_id` 与当前用户不匹配的强拦截**：
   在某些提交接口（如周排查）中，后端对 `inspector_id` 做了非常死板的强校验：

   ```python
   allowed_ids = {
       str(current_user.id),
       current_user.username,
       f"user-regulator-{current_user.username}",
   }
   if request.inspector_id not in allowed_ids:
       raise HTTPException(40300, "inspector_id与当前用户不匹配")
   ```

   如果前端传入的 `inspector_id` 是前端从列表中取出的真实用户 ID，而不是当前登录账号的 ID，就会直接报 `403`。

**建议后端修改：**

1. 请排查后端基于 `RBAC` 的权限查询服务（`get_permissions_for_user`），确保角色和权限之间的多对多关系能够被正确加载。
2. 建议放宽对 `inspector_id` 的强一致性校验（特别是当登录账号是管理员角色或监管端时，允许代填）。或者在详情接口明确下发应该提交的 `inspector_id`，以避免前后端对接不一致。

## 3. 周排查与联合巡检审核提交接口抛出 `500 Internal Server Error`

**现象：**
前端在“待审核”详情页点击“完成审核”提交时，后端报错 `500 Internal Server Error`。
后端日志打印如下错误：

```python
File "D:\Safefood-Platform\backend\app\modules\inspection\api.py", line 1517, in audit_weekly_task   
  "status": task.status.value, 
            ^^^^^^^^^^^^^^^^^ 
AttributeError: 'str' object has no attribute 'value'
```

**原因分析：**
在 `api.py` 的 `audit_weekly_task` 接口（约 1517 行）和 `audit_joint_task` 接口（约 1401 行）返回值构造中，使用了 `task.status.value`。但在某些数据库环境（如 SQLite）下，SQLAlchemy 查询出的 Enum 字段 `task.status` 可能直接被解析为字符串（`str`），而不是枚举对象，导致访问 `.value` 属性时抛出 `AttributeError`。

**建议后端修改：**
可以参考其他接口（如 `get_weekly_task_detail` 接口中约 644 行）的做法，使用安全的方式获取状态值：

```python
# 原代码：
"status": task.status.value,

# 修改为安全获取：
"status": task.status.value if hasattr(task.status, "value") else str(task.status),
```

## 4. 数据库布尔值序列化类型不一致（日管控等模块）

**现象：**
在执行端查询日管控详情接口 `GET /daily-controls/tasks/{id}` 时，前端期望返回的 `is_qualified` 为标准的布尔值（`true` 或 `false`），但接口实际返回的值类型极不稳定：有时是整型 `0` 或 `1`，有时是字符串 `'false'` 或 `'true'`。这导致前端依赖强类型（`=== false`）的列表过滤逻辑失效，出现明明有不合格项，但“待整改”列表却渲染为空的 Bug。

**原因分析：**
在某些数据库（如 SQLite）中，并没有原生的 Boolean 类型，布尔值在底层会被存储为 `0/1` 或 `VARCHAR`。SQLAlchemy 查询并组装 JSON 响应时，如果没有显式进行布尔类型强转（Type Casting），或者在 Pydantic 模型中没有严格定义字段类型，就会将底层数据库的原生值直接透传给前端，导致类型混乱。

**建议后端修改：**

1. 在 `api.py` 中组装 `form_snapshot` 时（例如 `item_data["is_qualified"] = result.is_qualified`），建议显式处理类型：

```python
item_data["is_qualified"] = True if result.is_qualified in (1, '1', 'true', True) else False
```

2. 或者确保对应的 Pydantic Schema 中将该字段严格定义为 `bool`，以便 FastAPI 在响应序列化时自动完成纠正。
