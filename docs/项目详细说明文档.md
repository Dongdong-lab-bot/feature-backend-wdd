# Safefood Agent Cluster 项目详细说明文档

## 📋 文档概述

| 项目 | 内容 |
|------|------|
| 项目名称 | 食安大模型 Agent 集群 (Safefood Agent Cluster) |
| 项目定位 | 企业级食品安全 AI 系统的核心大脑集群 |

## 1. 项目总体架构

### 1.1 架构设计理念

```
┌─────────────────────────────────────────────────────────────────┐
│                        边缘端 (EdgeAgent)                        │
│                   YOLO 视觉算法 → 告警推送                        │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTP POST /api/v1/alerts
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                         基建网关层                                │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐              │
│  │  告警接口   │  │  聊天接口   │  │  权限校验   │              │
│  │ alert.py   │  │  chat.py    │  │  JWT验证   │              │
│  └─────────────┘  └─────────────┘  └─────────────┘              │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                      消息总线 (Redis Streams)                     │
│              agent_message_bus | ALERT_RECEIVED 等事件           │
└───────────────────────────┬─────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ 监管 Agent    │   │  数据 Agent   │   │  路由 Agent   │
│ (Triage)      │   │   (Data)      │   │  (Router)     │
│               │   │               │   │               │
│ • VLM 校验    │   │ • Text-to-SQL │   │ • 意图识别    │
│ • 时空去重    │   │ • 数据分析    │   │ • 任务分发    │
│ • 违规判定    │   │ • 报表生成    │   │ • 对话管理    │
└───────────────┘   └───────────────┘   └───────────────┘
        │                   │                   │
        └───────────────────┼───────────────────┘
                            ▼
                    ┌───────────────┐
                    │  前端 (WebSocket) │
                    └───────────────┘
```

### 1.2 数据流向

1. **告警流程**：EdgeAgent → 网关 → 消息总线 → 监管 Agent → 数据/路由 Agent
2. **聊天流程**：用户 → 前端 → 网关 → 路由 Agent → 分发至专项 Agent → 汇总返回
3. **通知流程**：各 Agent → 消息总线 → 网关 → WebSocket/邮件推送

## 3. 核心模块详解

### 3.1 src/core/schemas.py — 全局数据宪法

这是项目中最重要的文件，所有外部输入/输出必须符合此模块定义的结构。

**核心枚举**：
| 枚举类 | 用途 |
|--------|------|
| `UserRoleEnum` | 用户角色 (store_manager/area_supervisor/enterprise_admin) |
| `AlertLevelEnum` | 告警级别 (high/medium/low/critical/unknown) |
| `VerifyResultEnum` | VLM校验结果 (true_violation/false_alarm/uncertain) |
| `ViolationTypeEnum` | 违规类型 (A01-A05, A99, A00) |
| `UserIntentEnum` | 用户意图 (query_summary/query_detail/export_report/...) |
| `AgentEventTypeEnum` | Agent间通信事件类型 |

**核心工具函数**：
- `beijing_now()`: 返回带北京时区的 Aware Datetime，禁止直接使用 datetime.now()
- `generate_id(prefix, random_length)`: 生成全局唯一ID，格式 `{PREFIX}{13位毫秒时间戳}{随机字母数字}`

### 3.2 src/core/config.py — 配置管理

基于 pydantic-settings 实现分层配置优先级：`.env` 文件 → 环境变量 → 默认值

**配置分组**：
- `ProjectSettings`: 项目名称、环境、调试模式、端口
- `MySQLSettings`: 主机、端口、数据库、管理员/只读账号
- `RedisSettings`: 主机、端口、密码、Stream 主题
- `LLMSettings`: API地址、密钥、模型名、超时、重试次数

**安全约束**：
- 所有密码字段使用 `repr=False`，防止日志打印泄露
- `llm_base_url` 必须以 `http://` 或 `https://` 开头

### 3.3 src/core/logger.py — 结构化日志

**核心特性**：
- 基于 loguru 开发
- 全链路 trace_id 关联（通过 contextvars 实现线程/异步隔离）
- 敏感信息脱敏（password/api_key/secret/token 等字段自动掩码）
- 自定义 AUDIT 日志级别（no=25）
- 分级存储：runtime/audit/error 三类日志文件

**日志级别与文件对应**：
| 级别 | 文件 | 保留期 |
|------|------|--------|
| DEBUG/INFO | runtime_{date}.log | 30 days |
| AUDIT | audit_{date}.log | 365 days |
| ERROR | error_{date}.log | 90 days |

### 3.4 src/core/exceptions.py — 异常体系

统一异常基类 `ProjectBaseException`，错误码按领域分组：

| 前缀 | 领域 | 示例 |
|------|------|------|
| 1xxx | 通用参数/配置 | PARAM_ERROR, CONFIG_ERROR |
| 2xxx | 大模型相关 | LLM_CALL_FAILED, LLM_FUSE_TRIGGERED |
| 3xxx | 安全/权限 | SQL_SECURITY_BLOCKED, PERMISSION_DENIED |
| 4xxx | 数据库/存储 | DB_CONNECT_FAILED, DB_EXECUTE_FAILED |
| 5xxx | Agent业务 | AGENT_INIT_FAILED, AGENT_HANDLE_FAILED |
| 6xxx | 路由Agent | INTENT_RECOGNITION_FAILED, ROUTE_DISPATCH_FAILED |
| 7xxx | 监管Agent | VLM_VERIFY_FAILED, ALERT_DUPLICATE |
| 8xxx | 基建组 | GATEWAY_REQUEST_INVALID, STORAGE_FAILED |

### 3.5 src/core/message_bus.py — 消息总线

**设计原则**：
- 业务层只依赖 MessageBus 抽象，不依赖具体实现
- 当前基于 Redis Streams 实现，预留 Kafka/RabbitMQ 扩展空间

**核心接口**：
- `start()` / `stop()`: 生命周期管理
- `send_message(topic, data)`: 发送消息，返回消息ID
- `subscribe_topic(topic, callback, group, consumer)`: 订阅消息

**消费者组模式**：
- 自动创建消费者组（若不存在）
- 使用 xreadgroup 读取，支持负载均衡与故障转移
- 回调成功后自动 ACK

### 3.6 src/agents/base_agent.py — Agent 抽象基类

**元类自动注册机制**：
- 继承 BaseAgent 的子类自动注册到 AgentRegistry
- 无需手动调用 registry.register()

**生命周期**：
```
__init__ → run() → [handle_event() ...] → stop()
```

**熔断器机制**：
- CLOSED → 正常，请求通过
- OPEN → 熔断，请求直接失败
- HALF_OPEN → 半开，允许少量试探请求

**配置参数**（子类可覆盖）：
- `FUSE_FAILURE_THRESHOLD = 5` — 触发熔断的连续失败次数
- `FUSE_RECOVERY_TIMEOUT = 60.0` — 熔断恢复等待时间（秒）
- `FUSE_HALF_OPEN_MAX_CALLS = 3` — 半开状态最大试探次数

## 4. API 接口规范

### 4.1 告警推送接口

**Endpoint**: `POST /api/v1/alerts`

**认证**: `X-API-KEY: <EDGE_AGENT_API_KEY>`

**Content-Type**: `multipart/form-data`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| alert_data | string (JSON) | 是 | EdgeAlertPayload 序列化字符串 |
| file | file (JPEG/PNG) | 否 | 带检测框标注截图，≤5MB |

**处理流程**：
1. 校验 X-API-KEY
2. 解析 multipart，提取 alert_data 和 file
3. 文件安全校验（魔数、格式白名单、分辨率、大小）
4. 校验 alert_data 字段完整性
5. 保存截图到 `/app/alert_images/{uuid}.jpg`
6. 时间转换：北京 naive → 带时区 aware datetime
7. level 映射为 AlertLevelEnum
8. 生成告警 ID：`ALT{timestamp}{random}`
9. 入库 AlertCoreModel
10. 写入 Redis Stream，触发 ALERT_RECEIVED 事件
11. WebSocket 广播前端

### 4.2 聊天交互接口

**Endpoint**: `POST /api/v1/chat/messages`

**认证**: `Authorization: Bearer <JWT>`

**处理流程**：
1. 校验 JWT 与 session_id 归属
2. 更新会话上下文
3. 路由 Agent 意图识别
4. 若置信度不足 → 返回 clarification
5. 若需要二次确认 → 生成 ConfirmationModel
6. 路由到目标 Agent
7. 聚合结果返回

**超时设置**：
- 整体接口超时：30 秒
- 超时返回 code=5002

## 7. 安全红线

| 红线 | 说明 |
|------|------|
| Schema 契约 | 所有输入/输出必须符合 `schemas.py` 定义的结构 |
| 只读安全 | 给 LLM 提供数据库 Tool，必须绑定只读账号 |
| 服务降级 | 大模型调用必须配置超时重试，失败3次必须返回兜底回复 |
| 全链路可追溯 | 关键决策必须写入审计日志 (log.audit()) |
| 异常规范 | 禁止裸抛 Exception，必须抛 ProjectBaseException 子类 |
| 敏感信息 | 禁止硬编码密钥/密码，密码字段使用 repr=False |

## 10. 常见问题

**Q: 如何新增一个业务 Agent？**
A: 继承 `BaseAgent`，实现 `run()` 和 `_on_event()` 方法，元类自动注册。

**Q: 如何修改 schemas.py？**
A: 需要架构师审批，通过 PR 合入。

**Q: 如何添加新的错误码？**
A: 在 `exceptions.py` 的 `ErrorCodeEnum` 中添加，遵循 4位数字前缀规则。

**Q: 如何调试日志？**
A: 设置 `.env` 中 `DEBUG=true`，AUDIT 日志会输出到控制台。
