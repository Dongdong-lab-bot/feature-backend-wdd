# 核心架构动作

1. 搭建统一边缘数据接入网关，标准化对接所有边缘盒子的 YOLO 告警 / 视频流；
2. 制定全局统一的 API 接口规范、数据字段标准、日志规范、权限体系；
3. 每个子 Agent 做成可插拔的标准化组件，预留集群调度接口
4. 统一的数据结构。无论子Agent怎么写，必须规定好统一的输入输出格式（例如所有的告警输入必须是标准的JSON，包含timestamp, location, violation_type, image_url, confidence等字段）。
5. 解耦。子Agent只做一件事，不要让一个Agent既负责查法规，又负责发通知。

## 监管Agent（研判Agent）

### 核心职责

当某一时间告警次数很多，超出设定阈值时，Agent介入。大概逻辑：yolo发生告警后，将yolo识别到的图片传到云端模型进行二次确认，确认无误再进入正常告警消息。

### 优化建议

* **触发逻辑优化：双轨制触发，覆盖全风险等级**
  放弃仅「超阈值才介入」的单逻辑，改为合规优先级驱动的双触发机制，完全匹配食安监管要求：
  * 高风险违规（抽烟、鼠患、生熟混放、从业人员赤膊、垃圾桶未加盖等）： **单条告警即触发云端二次校验** ，零容忍，避免单次严重违规漏检造成的监管事故；
  * 低风险违规（口罩佩戴不规范、工帽轻微滑落、物品乱摆等）： **单位时间内告警超阈值触发批量二次校验** ，平衡算力与效率，同时解决告警风暴问题。
* **核心职责补全，从「校验工具」升级为「合规监管中枢」**
  保留二次校验核心能力，新增食安场景必备的闭环能力：* 违规合规分级与条款绑定：二次校验时，不仅确认是否违规，还要按《餐饮服务食品安全操作规范》及门店属地监管细则，完成违规严重度分级，标注对应的合规条款，为后续台账生成、监管上报提供法定依据；
* * 告警聚合与风暴抑制：对同一摄像头、同一门店、同一类型的高频重复告警做聚合处理，比如 10 分钟内 10 次未戴口罩告警，聚合为 1 条「持续违规事件」，避免给用户推送海量重复消息；
  * 违规溯源与上下文关联：二次确认违规后，自动关联该门店的历史违规记录、对应责任人、摄像头点位信息，同步给数据处理 Agent 完成台账写入，无需人工干预；
  * 整改核验闭环：对接整改工单流程，对用户提交的整改图片 / 视频做自动核验，确认整改是否到位，更新违规事件状态。
* **技术架构优化，适配边缘 - 云端协同架构**
  边缘端先做初筛，仅推送疑似违规的关键帧 + 结构化检测结果到云端，而非全量图片，大幅节省带宽和云端算力；同时预留多模型扩展接口，后续可新增不同违规类型的专项校验模型，无需重构核心逻辑
* **建议的逻辑优化（抽帧与防并发）：**
  边缘盒子容易产生“告警风暴”（同一个人未戴口罩，3秒内YOLO可能推上来20张图）。
  * **第一步：** **Agent介入后，首先做**“时间/空间去重”**（比如同一摄像头、同一类型告警，1分钟内只取一张置信度最高的图）。**
  * **第二步：** **将去重后的图片发给云端VLM（如GPT-4o或通义千问VL），要求VLM输出JSON（包含：是否确认违规、违规特征描述）。**
  * **第三步：** **确认无误后，再下发正式告警。**
* **落地避坑红线**
  1. **可追溯性强制要求** ：二次校验的全流程（模型版本、校验时间、判定依据、置信度）必须全链路留痕存证，符合食安监管的可追溯要求，严禁黑盒无依据判定；
  2. **服务降级机制** ：云端服务异常时，自动降级为边缘端规则校验，不中断核心告警流程，避免监管失效；
  3. **合规边界** ：仅做违规事实判定，严禁给出突破食安法规的处置建议，所有判定规则必须有明确的法定依据。

## 数据处理Agent（数据分析Agent）

### 核心职责

对数据库进行管理和处理数据库交互的Agent。涉及按需从数据库中获取相关信息或者按需写入相关信息

### 优化建议

* **警惕“为了Agent而Agent”：** **如果只是单纯的“把确认的告警写入数据库”，或者“按固定条件查询数据”，这属于**确定性业务逻辑**，用传统的代码（API、CRUD函数）来做最快、最稳定、最便宜。大模型做固定格式的写入反而有幻觉风险。**
* **大模型的真正价值在于 Text-to-SQL（自然语言转查询）：**
  这个Agent的核心职责应该是处理**“非结构化需求”**。
  例如，不要让它管单纯的写入，而是让它负责高级查询：
  * **当需要统计时，把数据库的表结构（Schema）喂给它。**
  * **当收到问题“上个月哪个门店的未戴口罩违规最严重？”时，该Agent能自动写出SQL，去数据库查出结果，并生成一段分析结论，甚至画个图表。**
* **定位调整：** **把它定位为整个食安系统的**“数据分析师”**，而不仅是“数据库搬运工”。**
* **落地避坑红线**
  1. **权限最小化原则** ：给每个 Agent 分配独立的数据库账号，仅开放必要的读写权限，比如交互引导 Agent 仅开放读权限，监管 Agent 仅开放告警 / 整改数据的读写权限，严禁给任何 Agent 开放全量库表的管理员权限；
  2. **操作全链路审计** ：所有数据库操作（哪个 Agent、什么时间、执行了什么操作、修改了什么数据）必须全量留痕，生成审计日志，满足监管部门的审计要求；
  3. **核心数据容灾** ：对合规台账、违规记录、整改凭证等核心监管数据，必须做异地多副本备份，严禁数据丢失。

## 交互引导Agent（中枢路由Agent）

### 核心职责

面向用户的。负责与用户的沟通，并根据用户业务需求提供相关建议。例如：用户询问今日告警情况，此Agent在回复问题之外，还能够加一句“需要我将违规人员列出来吗"。

### 优化建议

* **它是用户唯一的入口：** **在Agent集群中，用户不需要知道“监管Agent”和“数据Agent”的存在。用户只和这个“交互引导Agent”聊天。**
* **它的隐藏身份是“路由（Router）”：**
  * **当用户说“给我看看今天抓拍的未戴口罩图片”，**交互Agent**理解意图后，会在后台调用**数据Agent**去查数据。**
  * **当收到新的告警事件时，它可以主动给用户发消息：“老板，刚刚后厨发生一起严重违规，我已经拦截并核实了，是否查看？”**
* **配置长期记忆（Memory）：** **记住用户的偏好。比如用户每次看完告警汇总都习惯问怎么处罚，这个Agent学习到之后，下次就可以直接附带相关的处罚建议。**
* **核心能力升级，从「对话机器人」升级为「业务闭环入口」**
  * 多角色自适应交互：针对食安系统不同用户角色的核心需求，做差异化的交互与引导，彻底避免通用化话术：
    * 门店店长：聚焦待办整改、今日违规、合规要求，主动推送待整改工单，引导完成整改闭环；
    * 区域督导：聚焦管辖门店合规排名、高频违规、整改率、风险门店预警，引导下沉巡检与专项整改；
    * 企业负责人：聚焦全公司合规大盘、监管风险、月度合规报表，引导管理优化与风险防控；
  * 全流程主动引导，而非仅问答后补充追问：打破仅用户提问后才引导的限制，基于业务事件主动触发服务，比如：
    * 用户登录系统时，主动推送「今日有 3 条待整改违规，剩余整改时限 2 小时，是否立即查看？」；
    * 整改工单超时时，主动推送「XX 门店违规整改已超时，是否立即联系门店负责人？」；
    * 门店出现高频重复违规时，主动推送「XX 门店本周未戴口罩违规占比 60%，需要我生成针对性的员工培训方案吗？」；
  * 跨 Agent 业务联动：不止是回答问题，可直接联动其他 Agent 完成用户指令，形成闭环。比如用户说「把今天的违规台账发给我」，自动调用数据处理 Agent 拉取数据、生成合规台账，直接发送给用户；用户说「给 XX 门店发整改通知」，自动调用监管 Agent 生成标准化整改工单，推送给对应责任人。
* **引导逻辑优化，精准不骚扰**
  建立引导规则体系，同一个场景不重复引导，对已完成闭环的事件不再推送引导信息，避免用户反感；同时设置引导优先级，高风险事件优先推送，低风险事件仅在用户空闲时段推送。
* **落地避坑红线**
  1. **操作可确认原则** ：所有涉及数据修改、通知推送、工单下发的操作，必须经过用户二次确认，严禁自动执行，避免误操作造成的管理事故；
  2. **话术合规性** ：所有引导话术、业务建议必须符合食安监管法规，严禁给出违规的操作建议，所有管理建议必须有明确的合规依据；
  3. **服务降级机制** ：大模型服务异常时，自动降级为固定菜单式交互，保证核心查询、工单处理功能可用，不影响用户核心业务操作。
