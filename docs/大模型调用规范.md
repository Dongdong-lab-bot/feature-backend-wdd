# 大模型调用规范

> **版本**：V1.0
> **生效日期**：2026-04-30
> **适用范围**：全项目所有 LLM/VLM 调用
> **核心约束**：稳定性红线 — 所有大模型调用必须配置超时重试、熔断降级逻辑，连续3次调用失败必须切硬编码兜底

---

## 1. 总则

### 1.1 设计目标

- 统一全项目大模型调用入口，禁止业务组私自实例化 OpenAI 客户端或硬编码 API 密钥
- 内置超时控制、指数退避重试、熔断降级，确保大模型服务异常时不阻塞业务流
- 自动审计日志埋点，满足可追溯红线要求

### 1.2 调用入口

| 组件 | 文件位置 | 职责 |
|------|----------|------|
| 异步 LLM 客户端 | `src/core/llm_client.py` | 通用文本对话（通义千问、GPT-4o 等） |
| 异步 VLM 客户端 | `src/core/vlm_client.py` | 视觉理解（告警图片二次校验） |
| 熔断器管理器 | `src/core/circuit_breaker.py` | 跨客户端复用的熔断状态机 |

业务代码禁止直接 `import openai` 或构造 `AsyncOpenAI` 实例，统一通过 `AsyncLLMClient` 调用。

---

## 2. 配置管理

所有大模型配置通过 `src/core/config.py` 的 `LLMSettings` 读取，优先级：`.env` > 环境变量 > 默认值。

| 配置项 | 环境变量 | 默认值 | 说明 |
|--------|----------|--------|------|
| 基础 URL | `LLM_BASE_URL` | — | 必须以 `http://` 或 `https://` 开头 |
| API 密钥 | `LLM_API_KEY` | — | `repr=False`，禁止日志打印 |
| 模型名称 | `LLM_MODEL_NAME` | — | 如 `qwen-max`、`gpt-4o` |
| 超时时间 | `LLM_TIMEOUT` | 30 | 单位：秒 |
| 最大重试 | `LLM_MAX_RETRIES` | 3 | 不含首次尝试 |

## 3. 调用规范

### 3.1 标准调用方式

```python
from src.core.llm_client import AsyncLLMClient

async with AsyncLLMClient() as client:
    result = await client.chat(
        messages=[{"role": "user", "content": "..."}],
        trace_id="trace-xxx",
    )
```

### 3.2 参数说明

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `messages` | list | 是 | OpenAI 标准消息格式 |
| `trace_id` | str | 是 | 链路追踪 ID，用于审计日志关联 |
| `provider_id` | str | 否 | 模型提供者标识，默认 `default` |
| `temperature` | float | 否 | 采样温度，默认 0.7 |
| `max_tokens` | int | 否 | 最大生成 token 数 |

### 3.3 异常处理

| 异常类型 | 触发条件 | 业务处理建议 |
|----------|----------|--------------|
| `LLMFuseTriggeredError` | 熔断器开启 | 直接返回兜底文本，不再重试 |
| `LLMResponseInvalidError` | 响应格式异常（空 choices / null content） | 记录错误，返回兜底文本 |
| `LLMCallFailedError` | 重试耗尽后仍失败 | 记录错误，返回兜底文本 |

### 3.4 兜底响应

连续失败触发熔断或重试耗尽时，必须返回结构化兜底响应：

```python
"抱歉，当前智能分析服务繁忙，请稍后再试。"
```

禁止向上抛裸异常导致链路中断。

## 4. 重试策略

### 4.1 重试范围

默认重试异常：`APIError`、`RateLimitError`、`APITimeoutError`、`APIConnectionError`

**不重试**：`AuthenticationError`（401）、`BadRequestError`（400）、业务逻辑异常

### 4.2 指数退避

| 尝试次数 | 延迟（秒） |
|----------|------------|
| 第 1 次失败 | 1.0 |
| 第 2 次失败 | 2.0 |
| 第 3 次失败 | 4.0 |
| 上限 | 10.0 |

---

## 5. 熔断策略

### 5.1 状态机

```
CLOSED（正常） --失败达阈值--> OPEN（熔断） --超时恢复--> HALF_OPEN（试探） --成功--> CLOSED
                                              |
                                              +--失败--> OPEN
```

### 5.2 默认参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 失败阈值 | 3 | 连续失败次数达到阈值后熔断 |
| 恢复超时 | 60 秒 | OPEN 状态持续多久后进入 HALF_OPEN |
| 半开试探 | 3 | HALF_OPEN 期间最多允许通过的请求数 |

### 5.3 隔离策略

按 `provider_id` + `prefix` 隔离熔断器实例，不同模型、不同场景互不影响。

---

## 6. 审计日志

每次大模型调用自动记录审计日志，包含：

- 事件类型：`llm_call completed` / `llm_call failed`
- `trace_id`：链路追踪 ID
- `success`：是否成功
- `latency_ms`：耗时（毫秒）
- `input_data`：入参（已脱敏）
- `output_data`：出参摘要
- `error_code` / `error_message`：失败时记录

审计日志写入 `logs/audit_YYYY-MM-DD.log`，保留 365 天。

---

## 7. 禁止事项

1. **禁止业务组私自构造 `AsyncOpenAI` 实例**
2. **禁止在代码中硬编码 API 密钥**
3. **禁止裸抛 `Exception`，必须使用 `ProjectBaseException` 子类**
4. **禁止绕过熔断器直接调用底层 API**
