# 端侧鉴权 + 外部 IdP 对接完成说明

## 任务目标
让监管端与执行端具备统一的登录与鉴权能力，同时支持外部身份提供方登录，保证端侧能稳定获取系统访问凭证。

## 已完成内容
- 统一登录接口，返回 accessToken、refreshToken、userInfo（含 tenantId、roleType）
- 账号密码登录与 Mock IdP 登录均可用
- 刷新 token 可用，并具备单次有效与并发保护
- 旧 refresh 被复用时触发熔断，撤销当前用户的有效 refresh 与 access
- 端侧鉴权校验已生效，app_client 与角色匹配，token 内 appClient 与当前端一致性校验
- 外部 IdP 首次登录自动建本地账号并分配角色
- 外部 IdP 同邮箱自动关联已有账号，避免重复用户
- 权限校验入口已具备，用于端侧接口访问控制

## 端侧使用结果
- 监管端、执行端均可使用同一套登录与鉴权流程
- 前端只需要处理登录与携带凭证，权限由后端统一校验
- 已满足日常联调与验收需求

## 外部 IdP 登录结果
- 支持使用外部身份登录
- 外部身份首次登录会自动绑定本地账号
- 同邮箱会自动关联已有账号，避免重复账号
- 角色与端侧类型保持一致
- 禁止通过 IdP 自动创建监管端(REGULATOR)角色，防止权限滥用
- 强制校验邮箱验证状态，未验证邮箱（email_verified!=true）不予创建或关联

## 当前交付状态
该任务已可交付使用，端侧鉴权与外部 IdP 登录流程已跑通并可持续迭代增强

## 需要前端配合的必做项
- 统一请求封装：所有请求必须使用统一封装，自动携带 token 并处理 401
- 刷新失败跳转登录：refresh 失败时清理本地 token 并跳转登录页，避免死循环
- Mock 联调注意：
  - **正常登录流程**：必须显式传递 `email_verified: true`，否则会被安全策略拦截导致登录失败。
  - **安全测试流程**：可传递 `email_verified: false` (或不传) 来验证后端是否正确拒绝了未验证邮箱的登录请求。
