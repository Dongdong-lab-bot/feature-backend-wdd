# 智慧食安 - 数字化台账 API契约 (V2.1 最终开发标准)

## 一、全局规范 (Global Standards)

### 1.1 业务状态码 (Business Codes)

响应体中的 code 字段必须遵循下表，前端根据此 code 执行统一弹窗或跳转逻辑：

| 业务状态码 | HTTP状态码 | 描述           | 处理建议                     |
| ---------- | ---------- | -------------- | ---------------------------- |
| 20000      | 200        | 操作成功       | 无需处理/轻提示              |
| 40001      | 400        | 重复提交       | 提示“请勿重复操作”         |
| 40401      | 404        | 资源不存在     | 提示“台账或任务已失效”     |
| 40301      | 403        | 权限不足       | 拦截访问并提示“无权操作”   |
| 50000      | 500        | 服务器内部错误 | 提示“系统繁忙，请稍后再试” |

### 1.2 统一响应结构

所有接口必须返回以下固定结构：
**非分页响应 (示例)**：

```json
{
  "code": 20000,
  "msg": "success",
  "data": { "id": 1024, "status": "SIGNED" }
}
```

**分页响应 (示例)**：

```json
{
  "code": 20000,
  "msg": "success",
  "data": {
    "total": 105, // 总记录数
    "page": 1, // 当前页码
    "size": 10, // 每页数量
    "records": [ ... ] // 数据列表
  }
}
```

### 1.3 幂等性控制 (Idempotency)

对于**暂存 (Draft)**等写操作，前端必须在请求头携带 X-Request-ID。

- Header: `X-Request-ID: <uuid-v4>`
- 后端逻辑: 缓存 ID 5分钟，若收到重复 ID，直接返回上一次结果，不重复写库。

## 二、核心枚举定义 (Core Enums)

### 2.1 台账实例状态 (InstanceStatus)

| 枚举值   | 描述   | 触发场景                                 |
| -------- | ------ | ---------------------------------------- |
| PENDING  | 待填写 | SOP 任务自动生成，用户尚未操作           |
| FILLING  | 填写中 | 用户已录入部分数据并点击“暂存”         |
| SIGNED   | 已签名 | 用户提交签字，数据被 Hash 锁定，不可修改 |
| ARCHIVED | 已归档 | 历史备份数据 (只读)                      |

## 三、台账实例与填报 (Instance & Submission)

### 3.1 获取台账任务列表(分页)

- 请求方式：GET
- 接口地址：`/ledger/instances`
- Query 参数: `page`, `size`, `date` (YYYY-MM-DD), `status`
- 权限: 执行端拉取自己的，监管端按 `canteen_id` 过滤。

### 3.2 暂存台账 (支持幂等)

- 请求方式：PUT
- 接口地址：`/ledger/instances/{id}/draft`
- Header: `X-Request-ID` (必填)
- Body:

```json
{ "form_data": { "temp": "36.8", "remark": "暂存" } }
```

### 3.3 签字提交 (Sign & Submit)

- 请求方式：POST
- 接口地址：`/ledger/instances/{id}/submit`
- 核心逻辑: 状态变更为 SIGNED，后端计算 `security_hash` 防篡改。
- Body:

```json
{
  "form_data": { ... },
  "signature_image": "https://oss-url.com/sign_2026.png" // 不接受 Base64
}
```

## 四、模板与任务管理 (CRUD)

**权限**: 仅监管端可操作

### 4.1 模板管理 (Template)

- GET `/ledger/templates`: 分页列表
- POST `/ledger/templates`: 创建模板 (JSON Schema)
- PUT `/ledger/templates/{id}`: 更新模板 (不影响历史快照)
- DELETE `/ledger/templates/{id}`: 软删除

### 4.2 SOP 调度任务 (SOP Task)

- POST `/ledger/tasks`: 创建调度 (设置 Cron 表达式与派发范围)
- PUT `/ledger/tasks/{id}/status`: 启用/停用任务

## 五、安全与数据保障

### 5.1 验签接口 (Data Integrity)

- 请求方式：GET
- 接口地址：`/ledger/instances/{id}/verify`
- 用途: 比对当前内容与 `security_hash`，用于第三方审计校验。

### 5.2 历史追溯 (Snapshot)

- 注意: 后端在生成 Instance 时必须包含 `schema_snapshot`。
- 规则: 前端渲染时优先使用 Instance 自带的快照，而非 Template 实时数据。

## 六、完成率统计与报表导出 (Reporting)

本章节基于前文「全局规范」(统一响应结构、业务状态码、多租户隔离) 约定。

### 6.1 完成率统计(按食堂维度)

- 请求方式：GET
- 接口地址：`/report/completion`

#### 用途

- 食堂端管理员:查看「自己食堂」某一天的台账完成率。
- 区域/监管端管理员:查看「所管理食堂列表」当天的完成率概览。

#### 权限

建议使用「查看类」权限，而非导出权限，例如：`DASHBOARD_VIEW_REGULATOR` / `DASHBOARD_VIEW_CANTEEN` 等 VIEW 权限；与 6.2 `/report/export` 使用的 `DASHBOARD_EXPORT` 权限明确区分，符合最小权限原则。

#### Query 参数

| 参数名     | 类型   | 必填 | 说明                                                               |
| ---------- | ------ | ---- | ------------------------------------------------------------------ |
| date       | string | 是   | 统计日期，格式 YYYY-MM-DD，对应 BizLedgerInstance.create_date      |
| canteen_id | int    | 否   | 食堂 ID:不传则统计当前用户有权限看到的所有食堂; 传入则只统计该食堂 |

> 注：tenant_id 一律从登录上下文获取，不通过参数传入。

#### 后端校验规则

后端必须校验 `canteen_id` 是否属于当前用户可见的食堂范围，否则返回 40301 (权限不足)。

#### 响应结构

遵循统一响应规范：

```json
{
  "code": 20000,
  "msg": "success",
  "data": {
    "total": 120,
    "date": "2026-02-25",
    "completed": 90,
    "rate": 0.75,
    "items": [
      {
        "canteenId": 101,
        "total": 20,
        "completed": 18,
        "rate": 0.9
      },
      {
        "canteenId": 102,
        "total": 30,
        "completed": 24,
        "rate": 0.8
      }
    ]
  }
}
```

#### 字段说明

- date:本次统计的日期。
- total:在当前筛选条件下的台账实例总数(整体分母)。`<br>`注：此处的 total 是统计总数，不是分页 total。
- completed:其中 `status ∈ {SIGNED, ARCHIVED}` 的实例数(整体分子)。
- rate:完成率，`completed / total`，范围 0–1，保留 2 位小数，四舍五入。
- items:按食堂维度的明细列表:
  - canteenId:食堂 ID。
  - total / completed / rate:该食堂当天的完成情况，rate 同样保留 2 位小数。

#### 业务规则

1. 仅统计当前租户下、且当前用户有权限查看的食堂。
2. 完成状态定义为:SIGNED 或 ARCHIVED(与 2.1 台账状态枚举保持一致)。
3. 若 `total = 0`，`rate` 约定为 0.00。
4. 统计必须使用数据库聚合查询实现，按 `canteen_id` 分组，例如 `GROUP BY canteen_id`，严禁逐食堂循环发 N+1 查询。

#### 错误示例

- 参数格式错误(如 date 非法):

```json
{
  "code": 40002,
  "msg": "invalid date format, expect YYYY-MM-DD",
  "data": null
}
```

- canteen_id 超出当前用户可见范围:HTTP 403 + code = 40301。
- 服务器内部错误:HTTP 500 + code = 50000。

### 6.2 报表导出(Excel / PDF)

- 请求方式：GET
- 接口地址：`/report/export`

#### 用途

导出指定日期、按食堂维度的完成情况报表，用于监管端或食堂端留存。支持 Excel / PDF 两种格式。表头严格来源于实例的 `schema_snapshot`，保证与历史模板完全一致。

#### 权限

建议: `DASHBOARD_EXPORT`(导出专用权限，与查看权限分离)。

#### Query 参数

| 参数名      | 类型   | 必填 | 说明                                                                        |
| ----------- | ------ | ---- | --------------------------------------------------------------------------- |
| date        | string | 是   | 导出日期，格式 YYYY-MM-DD                                                   |
| format      | string | 是   | 导出格式: excel 或 pdf                                                      |
| template_id | int    | 是   | 台账模板 ID:单次导出仅支持单一模板，用于保证表头与 schema_snapshot 一一对应 |
| canteen_id  | int    | 否   | 食堂 ID:不传则导出当前用户有权限看到的所有食堂数据                          |
| status      | string | 否   | 台账状态过滤: PENDING/FILLING/SIGNED/ARCHIVED，不传则不过滤                 |

#### 入参校验规则

1. 如传入的 `status` 不在枚举范围内，后端必须返回 40002。
2. 如传入的 `canteen_id` 不在当前用户可见范围内，必须返回 40301。
3. 如筛选条件命中多模板而 `template_id` 未指定，后端应拒绝导出并返回业务错误 (例如 40004)。

#### 成功响应

- HTTP 状态码:200
- Content-Type:

  - Excel: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
  - PDF: `application/pdf`
- Content-Disposition(推荐命名规范):

  ```
  attachment; filename="tenant_{tenantId}_report_{date}.xlsx"
  ```

  或加上时间戳避免浏览器缓存冲突:
  ```
  tenant_{tenantId}_report_{date}_{timestamp}.xlsx
  ```

  PDF 同理。

#### 文件内容要求

1. **表头**:
   - 从当前导出范围内的实例的 `schema_snapshot` 中解析字段定义，按 schema 中约定的顺序生成列标题。
   - 允许在最前面追加固定列，例如:日期、食堂名称、模板名称、状态。
   - 单次导出仅支持单一模板(由 `template_id` 指定)，保证字段集合与该模板对应的 `schema_snapshot` 完全一致。
2. **数据行**:
   - 每一行对应一个 `BizLedgerInstance`。
   - 固定列(日期、食堂名称、模板名称、状态等)直接取实例字段。
   - 业务字段按表头顺序，从实例的 `content` 中取值填充。

#### 条数限制与性能要求

1. 导出的实例行数 **最多 1000 条**。
2. 后端必须使用物理限制，例如 `LIMIT 1001` 查询:
   - 若查询结果条数 > 1000，则拒绝导出并返回业务错误;
   - 严禁先查全量记录到内存再 `len(records)` 判断后截断。

#### 错误示例

- 超限错误:

```json
{
  "code": 40003,
  "msg": "too many records to export, limit is 1000",
  "data": null
}
```

- status 非法枚举值:

```json
{ 
  "code": 40002, 
  "msg": "invalid status value, expect one of PENDING/FILLING/SIGNED/ARCHIVED", 
  "data": null 
}
```

- 多模板冲突(未指定 template_id 却命中多模板):

```json
{
  "code": 40004,
  "msg": "export requires a single template, please specify template_id",
  "data": null
}
```

- 权限不足:HTTP 403 + code = 40301。
- 服务器内部错误:HTTP 500 + code = 50000。

#### 审计与安全

每一次导出行为必须记录操作日志，至少包含:

- 导出日期 date
- template_id
- 实例行数(实际导出条数)
- 导出时间戳

> 日志需保留以备事后审计与问题追踪。

## 💡 给开发者的特别提示

1. 分页强制性: 所有列表严禁全量拉取，必须走 limit/offset 物理分页。
2. 签名规范: 前端必须先调用 OSS 上传接口，将返回的 URL 传给本契约接口。
3. 数据一致性: 更新模板(PUT)时，必须重新生成新的 `schema_hash`。
